TEST SUMMARY
======================================================================

This test validates whether the POST /api/pay endpoint accepts arbitrary
wallet addresses without verifying they match the payment owner.

The vulnerability allows an attacker to:
1. Intercept a legitimate payment ID
2. Substitute their own wallet address
3. Generate a transaction that sends funds to the attacker's wallet
   instead of the legitimate customer's wallet

VULNERABILITY ANALYSIS
======================================================================

🚨 VULNERABILITY CONFIRMED! 🚨

The endpoint accepted an arbitrary wallet address that doesn't match
the payment owner. This is a critical authorization bypass vulnerability.

ATTACK SCENARIO:
1. Legitimate customer (wallet TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA) created and linked payment
2. Attacker intercepted the paymentId
3. Attacker successfully substituted their wallet (Vote111111111111111111111111111111111111111)
4. Transaction would send funds to attacker instead of legitimate customer

TRANSACTION DETAILS:
  Transaction generated: YES
  Transaction length: 360 characters
  Receipt PDA: 79WaRVuUSPiR5BUuKr63tJ8mdSPi83TqAaiJFDGbiW8j

SECURITY IMPACT:
- Severity: CRITICAL
- Attack Complexity: LOW (only requires payment ID)
- Financial Impact: HIGH (complete payment hijacking)
- User Impact: Direct financial loss

RECOMMENDATION:
The /api/pay endpoint MUST verify that the provided 'account' parameter
matches the wallet address associated with the payment's linked code.

Suggested Fix:
Before generating the transaction, query the payment data and verify:
  payment.walletPubkey === input.account

If they don't match, return 403 Forbidden with an appropriate error message.
